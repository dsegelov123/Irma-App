Section 1: Immutable Core Persona & System Rules
1.1 Core Identity
Irma is a privacy-first, smart, AI-driven period-tracker application designed to deliver highly personalized, long-term predictive insights by treating the user like an intelligent adult. The application completely rejects standard industry tropes—there are no intrusive, rigid onboarding medical exams, no patronizing corporate-clinical layouts, and no stereotypical "pink-tinted flower" calendars.
1.2 Telemetry & Analytics Isolation
* Third-party tracking, crash reporting, and UI analytics SDKs (e.g., Firebase Analytics, Sentry) are strictly allowed only for non-health lifecycle events.
* The Zero-Telemetry Rule: No user-entered text, period cycles, symptoms, biometrics, or predicted dates may ever be passed as parameters to any third-party analytics SDK. Analytics are limited strictly to system performance and UI navigation tracking (e.g., screen_view).
Section 2: Data Architecture & State Engine
2.1 Cryptographic Architecture (Privacy-First Cloud Sync)
To maintain absolute user privacy without losing cross-device data recovery, the application must enforce an End-to-End Encrypted (E2EE) Client-Side Encryption with Key Escrow workflow:
* Encryption Standard: All sensitive health data (symptom logs, cycle lengths, calendar dates, notes, chat transcripts) must be encrypted on the device before being sent to the cloud database using AES-256-GCM.
* Key Derivation & Escrow: The encryption key is cryptographically generated upon the user's first authentication via Google or Apple Sign-In.
* Cross-Device Recovery: The generated encryption key must be securely stored in a secure cloud Key Management Service (KMS) or hardware security module (HSM) linked uniquely to the validated OAuth UID (Apple/Google ID token). When logging into a new device, successful authentication pulls down the user's unique key to decrypt the incoming synced database blobs locally.
* The Server-Blind Rule: The cloud database houses only unreadable, encrypted cryptographic blobs. The database provider has zero visibility into the user's actual cycles, symptoms, or conversation histories.
2.2 Local Storage & History
* Chat History Retention: All message transcripts generated within the Smart Chat Interface must be stored permanently inside the client device's local encrypted database (e.g., encrypted SQLite or Hive) to allow the user to view past conversations.
* Sync Security: Local chat transcripts are subject to the same E2EE cloud synchronization rules outlined in Section 2.1.
2.3 Clinical Knowledge Architecture (NHS Guidance Grounding)
To anchor Irma’s advice in verified medical reality without bloating the local application binary size, the system must employ an optimized, lightweight knowledge strategy:
* Data Scope: The system must be thoroughly trained or grounded on official National Health Service (NHS) clinical documentation, guidelines, and symptom definitions regarding menstrual, gynecological, and systemic health.
* Storage Optimization (Hybrid Vector Architecture): To keep the final app installation size compact, the architecture must avoid storing thousands of raw text medical articles on-device. Instead, the agents must implement a compressed vector embeddings database (or a high-efficiency text index) representing structural NHS health concepts.
* Cloud Fallback Proxy: If a specific health query or predictive pattern requires deep clinical text that exceeds the optimized local index, the client may fetch the relevant verified NHS data segment securely from an external server on-demand, caching it locally in the client's encrypted sandbox temporarily.
2.4 Cloud AI LLM Parsing & Transit Security
Because the Smart Chat Interface utilizes an external cloud-based Large Language Model (LLM) to drive the Irma character persona, raw text input must transit to an AI endpoint for processing. To reconcile this with the application's privacy-first architecture, the agents must implement the following proxy pipeline:

2. Ovulation

7.2 Immediate State-Overriding & Reset Logic
While the engine dynamically learns over time using long-term trends, the architecture forces a hard event override when physical data is logged:
* The Early Logging Override: If the user logs the explicit start of a period via any input pipeline (Log Screen or Smart Chat), and this date occurs days before the engine's predicted window, the system must immediately halt current projections and forcefully adjust the active cycle calendar state to synchronize with the user's real-time physical telemetry.
* The New Cycle Anchor Rule: The logging of a period start date is the absolute, unyielding mathematical anchor that forces the initialization of a brand-new cycle timeline. It resets the phase duration counter back to Day 1.
7.3 Multi-Variable Pattern & Trigger Identification (Cross-Channel Mapping)
The application achieves its true predictive utility by systematically cross-referencing short-term behavioral trends against historical patterns and medical parameters from the NHS Knowledge Architecture to discover individual cross-channel symptoms.
* Cross-Channel Correlation Matrix: The state engine must automatically trace connections across three parallel data pipelines over continuous time:
1. Chronological Placement: The explicit phase index (e.g., Late Luteal).
2. External Triggers: Environmental or mental metrics flagged via chats or logs (e.g., a highly stressful work day).
3. Biometric Telemetry: Streamed variables parsed from native frameworks (e.g., Apple HealthKit signaling elevated stress/heart rate variability metrics on a trigger day).
* Predictive Symptom Modeling: The algorithm maps recurrent symptom chains (e.g., identifying that a user in their late luteal phase consistently develops migraines exactly 24 hours following an isolated spike in work-related stress or a wearable telemetry stress alert).
* Pre-Emptive Intervention Triggers: Once a pattern chain is statistically established, the system shifts from reactive logging to proactive modeling. The engine handles this via two pathways:
o Timeline Pre-Emption: Surfacing advice or notifying the user before a historical trigger window opens based purely on timeline progression.
o Live Telemetry Acceleration: Instantly launching a pre-emptive insight payload the moment wearable frame inputs (like high baseline stress) match an active trigger profile on a designated phase day.
7.4 Anomaly and Long-Term Trend Adaptation
The computation engine must differentiate between temporary physiological disruptions and permanent shifts in user cycle trends over time.
* The Outlier Detection Filter: If a completed cycle deviates drastically from the user's established mathematical variance baseline (e.g., a sudden, severe shift caused by illness, high-altitude travel, or acute medication), the system must explicitly flag that specific cycle as an outlier anomaly.
* Anomaly Processing Engine: While flagged as an outlier, the cycle's historical record is not deleted. The analytics engine must cross-reference that anomaly against logged textual inputs, chat contexts, and health metrics to try and isolate the underlying trigger factor (e.g., matching a 40-day cycle directly to an entry tagged "Severe Flu").
* The Rolling Average Exclusion: Outlier cycles are excluded from the primary rolling average algorithm to ensure a temporary illness does not permanently corrupt future baseline forecasts.
* Gradual Trend Drifting: If the user's cycle baseline changes permanently over a long timeline (e.g., cycles continuously lengthening or shortening over a sequence of months), the statistical engine must execute an adaptive window update. It must down-weight ancient historical baselines and dynamically re-calculate predictions to align precisely with the user's migrating long-term physiological trends.

Section 10: Tri-Metric (Body, Mind, Soul) Taxonomy & Analytics Engine
10.1 Structural Parameter Matrix
The tracking core organizes all personal health, cognitive, and emotional data vectors into an explicit hierarchical matrix. This taxonomy provides the foundational framework for pattern recognition, trigger modeling, and medical advice generation.
1. The Body Hierarchy (Physical Telemetry Baseline): Tracks the immediate physiological status of the user's machine.
o Sub-Metric A: Energy (Perceived bodily power and physical capacity).
o Sub-Metric B: Wakefulness (Alertness status, sleep quality impacts, and diurnal positioning).
o Sub-Metric C: Recovery (Physiological replenishment rate, muscle/tissue status, and somatic resilience).
2. The Mind Hierarchy (Mental Telemetry Baseline): Tracks executive function and cognitive performance parameters.
o Sub-Metric A: Focus (Attention span duration, concentration accuracy, and distraction resistance).
o Sub-Metric B: Creativity (Lateral thinking capacity, problem-solving expansiveness, and ideation fluidity).
o Sub-Metric C: Motivation (Task-initiation drive, executive execution velocity, and mental willpower thresholds).
3. The Soul Hierarchy (Emotional Telemetry Baseline): Tracks affective states, internal emotional architecture, and social capacity bounds.
o Sub-Metric A: Mood (Active emotional positioning and core internal state validation).
o Sub-Metric B: Social Bandwidth (Interpersonal capacity, communication thresholds, and external stimulus tolerance).
o Sub-Metric C: Stability (Emotional equilibrium maintenance, volatility damping rates, and grounding indices).
10.2 Mathematical Bound Mechanics & Score Computation
The state engine maps these distinct data points into unified numerical indicators to drive both the graphics framework and the diagnostic inference engine.
* Granular Numerical Scaling: Each sub-metric is modeled as a floating variable that resolves into a strict 0 to 100 value.
* The Equal-Weight Aggregation Framework: The composite parent metric score (Body, Mind, or Soul) is computed as an absolute mathematical value out of 100. Each of the three constituent sub-metrics contributes completely equally (exactly 33.3% each) to its respective parent category score. The core state calculation engine enforces a flat, unweighted average formula:
$$\text{Parent Score} = \frac{\text{SubMetric}_A + \text{SubMetric}_B + \text{SubMetric}_C}{3}$$
* Data Generation Architecture: These numeric scores are generated by cross-analyzing data processed from three simultaneous channels:
1. NHS Guidance: Structural baseline bounds, biological variance boundaries, and clinical definitions derived from the local reference core.
2. Recent and Historical Inputs: Tracking data captured from manual daily logs and continuous historical record arrays.
3. Smart Chat Analysis: Contextual indicators extracted via secure LLM parsing of user-initiated transcripts.
* Discrete Tier Triggers: For the purposes of visual graphic rendering and calculating predictive health forecasts, the continuous 0-100 range maps into three functional data outputs:
o Low Tier: Applied when composite scaling drops below a mathematically defined lower threshold.
o Medium Tier: Applied when composite scaling sits within stable middle equilibrium bounds.
o High Tier: Applied when composite scaling matches optimized peak status boundaries.
10.3 Predictive Application of the Tri-Metric Matrix
* Granular Personalization: By continuously monitoring the absolute positions and movement paths of these numeric scores across the monthly timeline, the system achieves an individual baseline matrix.
* Pre-Emptive Flare-Up Modeling: The predictive engine parses these sub-metric combinations to identify approaching physiological or emotional dips. If an uncharacteristic drop in one metric (e.g., Body/Recovery) reliably precedes a flare-up in another (e.g., Soul/Volatility or a physical menstrual symptom), the system identifies this pattern and dispatches targeted advice or schedules protective push notifications before the symptom window fully opens.
